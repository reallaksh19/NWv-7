## 0. Why this revision exists

The original plan treated **Weather, Market, Newspaper, Data Health, Buzz, and
Following** as roughly greenfield and proposed brand-new collectors and
`*_latest.json` outputs for each. The Phase 0 audit proves that is wrong for all
but one of them. Building to the original spec would have **forked five live
pipelines** and shipped JSON no page reads.

### Three live divergences the audit flagged

1. **Data Health drift (real bug).** `news_prefetch.yml` regenerates
   `quality_dashboard.json` and `insight_quality_report.json` every run but its
   `git add` stages neither. The committed dashboard is already a day staler
   than `sections_latest.json` from the *same* workflow. This is the original
   plan's "JSON not staged" hazard **already happening in `main`.**
2. **Main retain gap (real bug).** `fetch_sections_stories.py` drops stories at
   **24h**; `sectionsSnapshotFetcher.js` accepts **36h**. The 24–36h band can
   never be populated.
3. **Buzz is the one true gap.** `/tech-social` is served entirely by a
   client-side seed dataset; there is no static producer or JSON.

> **Mission reframe:** this is **~80% unify-and-fix, ~20% build-new**, not a
> greenfield buildout. The envelope/lifecycle/horizon machinery is the high
> value; the per-destination work is mostly *additive enrichment of files that
> already exist* plus *one* new layer (Buzz).

## 1. Standing guardrails (derived from the audit, non-negotiable)

These encode the original plan's "most dangerous mistakes" as hard rules, plus
what the audit exposed:

- **G1 — No schema bump without adapter support.** `sectionsSnapshotFetcher.js`
  *throws* `unsupported_snapshot_schema` for anything ≠ 1|2. Sections stays
  **schema 2**; the envelope is added as *optional fields*, never a v3 bump,
  until the adapter + validator are upgraded in their own slice.
- **G2 — Reuse filenames; never fork.** No `weather_latest.json`,
  `market_latest.json`, `newspaper_latest.json`, or `data_health_dashboard.json`
  beside the existing `weather_snapshot.json` / `market_snapshot.json` /
  `epaper_data.json` / `quality_dashboard.json`. Enrich in place.
- **G3 — Every committed output is staged, and the audit proves it.** Any new or
  regenerated committed JSON must be added to its workflow `git add`. **Exit
  gate for every PR: re-run the audit; the "committed output not staged"
  warning count must not increase (and should drop to 0).**
- **G4 — Envelope is additive.** Existing `*.cert.test.js` and `scripts/test_*`
  suites must stay green. Consumers keep working with or without new fields.
- **G5 — Lifecycle families are separate.** Weather forecasts, market snapshots,
  alerts, offers, and events get their own validity rules; **never** the
  `news_36h` rule.
- **G6 — Insight `stories[]` stays canonical.** `clusters[]` are additive; the
  adapter still reads the flat pool (`slotMeta` is explicitly display-unused).
- **G7 — Data Health reports, never gates.** It fails only on its own malformed
  JSON, never on a downstream WARN.
- **G8 — Repo test conventions.** Python tests → `scripts/test_*.py`; JS/UI →
  `scripts/test_*_static.mjs` and `src/**/*.cert.test.js`. **No new `tests/*.py`
  tree** (root `tests/` holds page verifiers, not units). Certification extends
  the **existing** `scripts/run_certification_gate.mjs` +
  `scripts/certification_manifest.json`, not a parallel suite.

## 2. Workstreams (re-scoped from the original 22 phases)

The original phase numbers are noted for traceability. Foundations are
unchanged and high-value; the per-destination phases are reconciled.

### WS-0 — Destination audit  *(orig Phase 0 — DONE in this PR)*
- `scripts/audit_destination_contracts.py` → `reports/destination_contract_baseline.{json,md}`
- `scripts/test_audit_destination_contracts.py` (repo convention)
- **Standing gate:** re-runnable; later wired into certification (WS-G) so drift
  is caught automatically.

### WS-A — Foundations *(orig Phases 1–5; KEEP AS DESIGNED, additive)*
The strongest part of the original plan. No destination forks; pure libraries.
- **A1 Universal envelope** — `scripts/static_json_contract.py` +
  `validate_static_json_contract.py`. Optional fields only (G1, G4).
- **A2 Content-class registry** — `scripts/content_classes.py` +
  `src/config/contentClasses.js`. Unknown class → validation failure.
- **A3 Lifecycle engine** — `scripts/content_lifecycle.py`. One function per
  family (news_36h, event_7d, planner, alert, offer, forecast, market). (G5)
- **A4 Seven-day horizon contract** — `scripts/horizon_policy.py` +
  `src/config/horizonPolicy.js` + `validate_7day_horizon.py`. Python/JS
  constants asserted equal by test. (UpAhead already obeys `[-1d,+7d]` /
  planner `[-6h,+7d]` — this makes it *explicit and testable*.)
- **A5 Source ledger + provider/freshness mix** — `scripts/source_ledger.py`.
  Mixes derived from **accepted** items, not configured feeds.
- **Exit:** all five import-clean; legacy fixtures still validate; zero behavior
  change. Tests: `scripts/test_static_json_contract.py`,
  `test_content_classes.py`, `test_content_lifecycle.py`,
  `test_7day_horizon_contract.py`, `test_source_ledger.py`.

### WS-B — Source resilience *(orig Phases 6–7)*
- **B1 Source-health degradation** — extend `scripts/source_health.py` (today it
  tracks `zeroItemRuns`/`status` but makes **no** fetch decision) +
  `scripts/source_policy_resolver.py`. Tier-A daily retry; reset on success.
- **B2 Provider client layer** — `scripts/news_provider_clients.py` wrapping the
  existing RSS / Google News / GDELT / TMDB fetch logic behind one tuple
  contract `(items, health, ledger_rows)`. Wrap **without** changing output
  shape; fixtures only, no live calls.
- **Exit:** degraded sources surface in the Data Health payload; Tier-A never
  permanently skipped. Tests: `test_source_policy_resolver.py`,
  `test_provider_clients.py`.

### WS-C — Feed enhancements *(orig Phases 8–9)*
- **C1 Sections 36h (fixes divergence #2).** `STORY_RETAIN_HOURS 24 → 36`;
  attach `contentClass` + `lifecycle` per story; broaden `SECTION_BACKFILL`
  (currently only `trichy→tn`, `muscat→world`); add envelope fields at
  **schema 2** (G1). Update `validate_sections_prefetch_output.py` with
  soft→strict thresholds. Update `sectionsSnapshotFetcher.js` + its cert test
  **only to read new optional fields**, not to require schema 3.
- **C2 Insight clusters (additive).** `scripts/build_insight_clusters.py` emits
  `clusters[]`/`angleCoverage` **alongside** `stories[]` (G6). UI may consume
  clusters when present; adapter still works without them.
- **Exit:** no visible ordinary story >36h; `stories[]` unchanged as canonical;
  adapters green. Tests: `test_sections_36h_gate.py`,
  `test_insight_cluster_builder.py`, `test_insight_quality_gate.py`.

### WS-D — Actionable horizon *(orig Phases 10–11)*
- **D1 UpAhead lifecycle.** Attach `lifecycle`/`horizon` objects + `contentClass`
  to `up_ahead.json` items using A3/A4; add `validate_upahead_quality_gate.py`.
  Keep the existing 7-bucket weekly plan.
- **D2 Planner static (the only genuinely new actionable output).**
  `scripts/build_planner_latest.py` → `public/data/planner_latest.json` from
  `up_ahead.json`, gated by `lifecycle.isActionValid && withinSevenDayHorizon`.
  Device-local saved planner overlay is preserved. **Stage it in
  `upahead_refresh.yml`** (which already `git add public/data/`, so this is
  covered — verify via audit, G3).
- **Exit:** every planner item resolves to a live UpAhead item within the
  horizon. Tests: `test_upahead_quality_gate.py`, `test_planner_static.py`.

### WS-E — Reconcile-in-place *(orig Phases 12, 14, 16, 18 — RESCOPED)*
**No new collectors, no new filenames.** Enrich the existing outputs and fix the
staging drift.
- **E1 Weather.** Add envelope + `weather_forecast`/`weather_alert` `lifecycle`
  to `weather_snapshot.json` inside `weather_snapshot_worker.py`. WeatherContext
  already reads it via `weatherService` — keep live `fetchWeather` as fallback
  only. **Do not** add `fetch_weather_static.py`.
- **E2 Market.** Add envelope + `market_snapshot`/`market_news_story`/
  `market_event` `lifecycle` and `validUntil` to `market_snapshot.json` inside
  `market_snapshot_worker.py`. **Do not** add `build_market_snapshot.py` or
  `market_latest.json`.
- **E3 Newspaper.** Extend `daily_brief.py` to emit edition metadata
  (`editionDate`, section→storyIds, source hashes) into `epaper_data.json`.
  **Do not** add `newspaper_latest.json`.
- **E4 Data Health (fixes divergence #1).** Extend `generate_quality_dashboard.mjs`
  to fold per-destination gate states + degraded sources into
  `quality_dashboard.json`, and **add `quality_dashboard.json` +
  `insight_quality_report.json` to the `news_prefetch.yml` `git add`** so the
  dashboard stops drifting (G3, G7). **Do not** add `data_health_dashboard.json`.
- **Exit:** envelopes present on all four; audit shows 0 "not staged" warnings;
  no `*_latest.json` siblings created. Tests:
  `scripts/test_weather_*_static.mjs` (extend), `test_market_static.py`,
  `test_newspaper_static.py`, `test_data_health_dashboard.py`.

### WS-F — New / optional layers *(orig Phases 15, 17)*
- **F1 Buzz (the one real new pipeline).** `scripts/build_buzz_latest.py` →
  `public/data/buzz_latest.json`, **shaped to feed the existing
  `buzzDataset.js` / `buzzSlo.js` contract** so `useTechSocialPageViewModel`
  consumes it without a rewrite. New workflow step (WS-G). 24h fresh / 36h max.
- **F2 Following/Topic static index (optional enrichment).**
  `scripts/build_following_latest.py` → `public/newsdata/following_latest.json` +
  `topic_detail_latest.json`, derived from migrated Sections/Insight; the page
  keeps deriving client-side if the file is absent. Reuses the existing
  `scripts/test_following_static.mjs` — **no `tests/test_following_static.py`.**
- **Exit:** Buzz renders from static when present, seed otherwise; Following
  unchanged when static absent. Tests: `test_buzz_static.py`, extend
  `test_following_static.mjs`.

### WS-G — Workflow integrity + certification *(orig Phases 20, 22)*
- **G-1 Staging correctness.** Reconcile every workflow `git add` against the
  audit's "expected static JSON" so all committed outputs (existing + new) are
  staged. Land the Data Health fix from E4 here if not already.
- **G-2 New workflow steps.** Insight clusters (C2) into `news_prefetch.yml`;
  Planner (D2) into `upahead_refresh.yml`; Buzz (F1) into a small refresh
  workflow (extend `weather_refresh.yml`/`market_refresh.yml` or add
  `buzz_refresh.yml`). Keep `STRICT_DATA_GATE: "false"` until each destination
  proves stable on real runs.
- **G-3 Certification extends, not replaces.** Add destination-contract +
  envelope + horizon assertions into the **existing**
  `scripts/run_certification_gate.mjs` / `certification_manifest.json`. Wire
  `audit_destination_contracts.py` in as a gate so future drift (new unstaged
  output, orphan route, forked filename) fails certification.
- **Exit:** `npm run test:certify` covers all destinations; audit warnings = 0.

## 5. Deep-review priority

1. **G3 staging proof** — re-run the audit on every PR; warnings must not grow.
2. **Lifecycle-family correctness** — weather/market/alerts ≠ news_36h.
3. **Schema compatibility** — no schema-3 sections; adapters stay green.

5. No forked `*_latest.json` beside existing snapshots (grep gate in cert).
6. Envelope is additive; cert tests unchanged-green.
7. Insight `clusters[]` never replaces `stories[]`.

## 6. The audit as a standing gate

`scripts/audit_destination_contracts.py` is re-runnable and read-only. Wiring it
into certification (WS-G-3) means the class of bug it just caught — a
regenerated-but-unstaged committed output — fails the build instead of silently
drifting in `main`. That is the durable payoff of doing Phase 0 first.
